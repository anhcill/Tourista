# 3.6.2. Thiết kế bảng User và các bảng liên quan

## 1. Giới thiệu

Phân hệ **Người dùng** trong hệ thống Tourista quản lý toàn bộ thông tin liên quan đến tài khoản người dùng, phân quyền, đánh giá, đặt chỗ và báo cáo. Phân hệ này bao gồm 5 bảng chính: `users`, `roles`, `user_roles`, `reviews` và `bookings`. Thiết kế hỗ trợ nhiều vai trò (ADMIN, USER, HOST) cho mỗi tài khoản, đánh giá đa tiêu chí cho tour/khách sạn, và theo dõi lịch sử đặt chỗ của từng khách hàng.

## 3. Chi tiết thiết kế từng bảng

### 3.1. Bảng `users`

| STT | Tên cột | Kiểu dữ liệu | Ràng buộc | Mô tả |
|-----|---------|--------------|-----------|--------|
| 1 | `id` | BIGINT UNSIGNED | **PK**, AUTO_INCREMENT | Khóa chính |
| 2 | `email` | VARCHAR(100) | **UNIQUE**, NOT NULL | Email đăng nhập |
| 3 | `password` | VARCHAR(255) | NOT NULL | Mật khẩu đã mã hóa (BCrypt) |
| 4 | `full_name` | VARCHAR(100) | NOT NULL | Họ tên đầy đủ |
| 5 | `phone_number` | VARCHAR(20) | **UNIQUE** | Số điện thoại |
| 6 | `avatar` | VARCHAR(500) | — | URL ảnh đại diện |
| 7 | `date_of_birth` | DATE | — | Ngày sinh |
| 8 | `gender` | ENUM('MALE','FEMALE','OTHER') | — | Giới tính |
| 9 | `is_active` | BOOLEAN | DEFAULT TRUE | Tài khoản có đang hoạt động không |
| 10 | `is_verified` | BOOLEAN | DEFAULT FALSE | Đã xác thực email chưa |
| 11 | `verification_token` | VARCHAR(255) | — | Token xác thực email |
| 12 | `token_expires_at` | DATETIME | — | Thời hạn token xác thực |
| 13 | `reset_password_token` | VARCHAR(255) | — | Token đặt lại mật khẩu |
| 14 | `reset_token_expires_at` | DATETIME | — | Thời hạn token đặt lại mật khẩu |
| 15 | `last_login_at` | DATETIME | — | Thời điểm đăng nhập gần nhất |
| 16 | `failed_login_attempts` | INT UNSIGNED | DEFAULT 0 | Số lần đăng nhập thất bại |
| 17 | `locked_until` | DATETIME | — | Khóa tài khoản đến thời điểm này |
| 18 | `created_at` | DATETIME | NOT NULL | Thời gian tạo tài khoản |
| 19 | `updated_at` | DATETIME | NOT NULL | Thời gian cập nhật gần nhất |

**Chỉ mục (Indexes):**
- `idx_users_email` trên `email` (UNIQUE)
- `idx_users_phone` trên `phone_number` (UNIQUE)
- `idx_users_is_active` trên `is_active`

**Chiến lược phân trang:** Sử dụng `BIGINT UNSIGNED` để hỗ trợ tới ~18 tỷ bản ghi.

**Cơ chế khóa tài khoản:** Khi `failed_login_attempts` >= 5, `locked_until` được set = NOW() + 30 phút. Mỗi lần đăng nhập thành công, `failed_login_attempts` reset về 0.

### 3.2. Bảng `roles`

| STT | Tên cột | Kiểu dữ liệu | Ràng buộc | Mô tả |
|-----|---------|--------------|-----------|--------|
| 1 | `id` | TINYINT UNSIGNED | **PK**, AUTO_INCREMENT | Khóa chính |
| 2 | `name` | VARCHAR(30) | **UNIQUE**, NOT NULL | Tên vai trò |
| 3 | `description` | VARCHAR(100) | — | Mô tả vai trò |

**Các vai trò hiện có:**

| id | name | description |
|----|------|-------------|
| 1 | ADMIN | Quản trị viên toàn hệ thống |
| 2 | USER | Người dùng thông thường (khách hàng) |
| 3 | HOST | Đối tác quản lý khách sạn/tour |

### 3.3. Bảng `user_roles` (Bảng trung gian)

| STT | Tên cột | Kiểu dữ liệu | Ràng buộc | Mô tả |
|-----|---------|--------------|-----------|--------|
| 1 | `user_id` | BIGINT UNSIGNED | **PK**, FK → users(id) ON DELETE CASCADE | Khóa ngoại đến users |
| 2 | `role_id` | TINYINT UNSIGNED | **PK**, FK → roles(id) ON DELETE CASCADE | Khóa ngoại đến roles |
| 3 | `assigned_at` | DATETIME | — | Thời gian gán vai trò |

**Khóa chính composite:** `(user_id, role_id)` — đảm bảo một user không bị gán trùng cùng một role.

**ON DELETE CASCADE:** Khi xóa user hoặc role, bản ghi liên kết tự động bị xóa theo.

### 3.4. Bảng `reviews`

| STT | Tên cột | Kiểu dữ liệu | Ràng buộc | Mô tả |
|-----|---------|--------------|-----------|--------|
| 1 | `id` | BIGINT | **PK**, AUTO_INCREMENT | Khóa chính |
| 2 | `user_id` | BIGINT | NOT NULL, FK → users(id) ON DELETE CASCADE | Người viết đánh giá |
| 3 | `booking_id` | BIGINT | FK → bookings(id) ON DELETE SET NULL | Đơn đã sử dụng dịch vụ |
| 4 | `target_type` | ENUM('HOTEL','TOUR') | NOT NULL | Loại đối tượng được đánh giá |
| 5 | `target_id` | BIGINT | NOT NULL | ID của hotel hoặc tour |
| 6 | `overall_rating` | INT | NOT NULL (1–5) | Điểm tổng quan |
| 7 | `title` | VARCHAR(200) | — | Tiêu đề đánh giá |
| 8 | `comment` | TEXT | — | Nội dung đánh giá chi tiết |
| 9 | `cleanliness` | INT | — | Điểm sạch sẽ (1–5) |
| 10 | `location` | INT | — | Điểm vị trí (1–5) |
| 11 | `service` | INT | — | Điểm dịch vụ (1–5) |
| 12 | `value_for_money` | INT | — | Điểm giá trị (1–5) |
| 13 | `guide_quality` | INT | — | Điểm hướng dẫn viên (1–5) |
| 14 | `organization` | INT | — | Điểm tổ chức (1–5) |
| 15 | `is_verified` | BOOLEAN | DEFAULT FALSE | Đã xác nhận đã sử dụng dịch vụ |
| 16 | `is_published` | BOOLEAN | DEFAULT TRUE | Đang công khai |
| 17 | `moderation_status` | ENUM | PENDING / APPROVED / REJECTED | Trạng thái kiểm duyệt |
| 18 | `admin_status` | ENUM | NOT NULL, DEFAULT PENDING | Trạng thái duyệt bởi admin |
| 19 | `rejection_reason` | VARCHAR(255) | — | Lý do từ chối duyệt |
| 20 | `helpful_count` | INT | DEFAULT 0 | Số người thấy hữu ích |
| 21 | `admin_reply` | TEXT | — | Phản hồi từ admin |
| 22 | `admin_replied_at` | DATETIME | — | Thời gian admin phản hồi |
| 23 | `partner_id` | BIGINT | FK → users(id) ON DELETE SET NULL | Đối tác phản hồi |
| 24 | `partner_reply` | TEXT | — | Phản hồi từ đối tác |
| 25 | `partner_replied_at` | DATETIME | — | Thời gian đối tác phản hồi |
| 26 | `created_at` | DATETIME | NOT NULL | Thời gian tạo |
| 27 | `updated_at` | DATETIME | NOT NULL | Thời gian cập nhật |

**Chỉ mục:**
- `idx_reviews_user_id` trên `user_id`
- `idx_reviews_target` trên `(target_type, target_id)` — tối ưu truy vấn đánh giá theo loại và ID
- `idx_reviews_booking_id` trên `booking_id`

**Đánh giá đa tiêu chí:** Tùy loại đối tượng (`target_type`), hệ thống sử dụng các tiêu chí khác nhau:
- **HOTEL:** cleanliness, location, service, value_for_money
- **TOUR:** guide_quality, organization, value_for_money, location, service

**Quy trình kiểm duyệt:** Khi user tạo review → `admin_status = PENDING` → Admin duyệt → `admin_status = APPROVED` hoặc `REJECTED` + `rejection_reason`. Review được gắn `booking_id` để xác nhận khách đã thực sự sử dụng dịch vụ → `is_verified = TRUE`.

## 4. Sơ đồ tuần tự — Đăng ký tài khoản

```
@startuml User_Registration_Sequence
!theme plain

title Biểu đồ trình tự — Đăng ký tài khoản

actor "Khách hàng" as user
participant "Frontend" as fe
participant "AuthController" as ctrl
participant "AuthService" as svc
participant "UserRepository" as repo
participant "EmailService" as email
participant "Database" as db

user -> fe : Nhấn "Đăng ký"\nĐiền form (email, mật khẩu, họ tên)
fe -> ctrl : POST /api/auth/register\n{email, password, fullName, phoneNumber}
activate ctrl

ctrl -> svc : register(request)
activate svc

svc -> repo : findByEmail(email)
activate repo
repo -> db : SELECT * FROM users\nWHERE email = ?
db --> repo : null / User record
deactivate repo

alt Email đã tồn tại
    svc --> ctrl : EmailAlreadyExistsException
    ctrl --> fe : 409 Conflict
    fe -> user : "Email đã được sử dụng.\nVui lòng đăng nhập hoặc\nkhôi phục mật khẩu"
else Email chưa tồn tại
    svc -> svc : Validate password\n(≥ 8 ký tự, chữ hoa, số, ký tự đặc biệt)

    alt Mật khẩu yếu
        svc --> ctrl : WeakPasswordException
        ctrl --> fe : 400 Bad Request
        fe -> user : "Mật khẩu phải có ít nhất 8 ký tự,\ngồm chữ hoa, số và ký tự đặc biệt"
    else Mật khẩu hợp lệ
        svc -> svc : Hash password\n(BCrypt, strength = 12)
        svc -> svc : Generate verification token\n(UUID v4, 64 ký tự)
        svc -> svc : Set token expires\n(NOW + 24 giờ)

        svc -> db : INSERT INTO users\n(email, password, full_name,\nphone_number, is_active,\nis_verified, verification_token,\ntoken_expires_at, created_at)
        activate db
        db --> svc : User record created\n(id = 1)
        deactivate db

        svc -> db : INSERT INTO user_roles\n(user_id, role_id = 'USER')
        db --> svc : Role assigned

        svc -> email : Gửi email xác thực
        activate email
        email -> user : Email: "Xác thực tài khoản\nTourista"\nNhấn link xác thực
        deactivate email

        svc --> ctrl : User registered\n(Chưa xác thực)
        deactivate svc

        ctrl --> fe : 201 Created
        fe -> user : "Đăng ký thành công!\nVui lòng kiểm tra email để\nxác thực tài khoản"
    end
end

== XÁC THỰC EMAIL ==

user -> fe : Nhấn link trong email\n(http://tourista.com/auth/verify?token=...)
fe -> ctrl : GET /api/auth/verify?token={token}
ctrl -> svc : verifyEmail(token)
activate svc

svc -> repo : findByVerificationToken(token)
activate repo
repo -> db : SELECT * FROM users\nWHERE verification_token = ?
db --> repo : User record / null
deactivate repo

alt Token không tồn tại hoặc đã hết hạn
    svc --> ctrl : InvalidTokenException
    ctrl --> fe : 400 Bad Request
    fe -> user : "Link xác thực không hợp lệ\nhoặc đã hết hạn.\nVui lòng yêu cầu gửi lại email"
else Token hợp lệ
    svc -> db : UPDATE users\nSET is_verified = TRUE,\nverification_token = NULL,\ntoken_expires_at = NULL,
updated_at = NOW()
    db --> svc : Updated

    svc --> ctrl : Email verified
    deactivate svc

    ctrl --> fe : 200 OK
    fe -> user : "Xác thực thành công!\nBạn có thể đăng nhập ngay"
end

@enduml
```

## 5. Sơ đồ tuần tự — Đăng nhập & Phân quyền

```
@startuml User_Login_Sequence
!theme plain

title Biểu đồ trình tự — Đăng nhập & Phân quyền

actor "Người dùng" as user
participant "Frontend" as fe
participant "AuthController" as ctrl
participant "AuthService" as svc
participant "JwtTokenProvider" as jwt
participant "UserRepository" as repo
participant "Database" as db

user -> fe : Nhấn "Đăng nhập"\nEmail + Mật khẩu
fe -> ctrl : POST /api/auth/login\n{email, password}
activate ctrl

ctrl -> svc : login(credentials)
activate svc

svc -> repo : findByEmail(email)
activate repo
repo -> db : SELECT * FROM users\nWHERE email = ?
db --> repo : User record / null
deactivate repo

alt Tài khoản không tồn tại
    svc --> ctrl : BadCredentialsException
    ctrl --> fe : 401 Unauthorized
    fe -> user : "Email hoặc mật khẩu không đúng"
else Tài khoản tồn tại
    svc -> svc : Check account lock\n(if locked_until > NOW → throw LockedException)

    alt Tài khoản bị khóa
        svc --> ctrl : AccountLockedException\n"Số lần đăng nhập thất bại quá nhiều.\nThử lại sau 30 phút"
        ctrl --> fe : 423 Locked
        fe -> user : "Tài khoản tạm khóa.\nThử lại sau 30 phút"
    else Tài khoản không bị khóa
        svc -> svc : Verify password\n(BCrypt check)
        alt Mật khẩu sai
            svc -> svc : failed_login_attempts + 1
            svc -> db : UPDATE users\nSET failed_login_attempts = N,\nlocked_until = NOW() + 30min\n(if N >= 5)
            db --> svc : Updated
            svc --> ctrl : BadCredentialsException
            ctrl --> fe : 401 Unauthorized
            fe -> user : "Email hoặc mật khẩu không đúng.\nSố lần còn lại: X"
        else Mật khẩu đúng
            alt Tài khoản chưa xác thực
                svc --> ctrl : EmailNotVerifiedException
                ctrl --> fe : 403 Forbidden
                fe -> user : "Tài khoản chưa xác thực.\nGửi lại email xác thực?"
            else Tài khoản đã xác thực
                svc -> svc : Reset failed_login_attempts = 0
                svc -> db : UPDATE users\nSET last_login_at = NOW(),\nfailed_login_attempts = 0,\nlocked_until = NULL
                db --> svc : Updated

                svc -> repo : findRolesByUserId(userId)
                activate repo
                repo -> db : SELECT r.* FROM roles r\nJOIN user_roles ur ON r.id = ur.role_id\nWHERE ur.user_id = ?
                db --> repo : List<Role> [USER, HOST]
                deactivate repo

                svc -> jwt : Generate tokens
                activate jwt
                jwt -> jwt : Create access token\n(userId, email, roles, exp=15min)
                jwt -> jwt : Create refresh token\n(userId, exp=7days)
                jwt --> svc : {accessToken, refreshToken}
                deactivate jwt

                svc --> ctrl : Login success + tokens
                deactivate svc

                ctrl --> fe : 200 OK\n{accessToken, refreshToken,\nuser: {id, email, fullName, roles}}
                fe -> user : Đăng nhập thành công\nChuyển hướng Dashboard
            end
        end
    end
end

== SỬ DỤNG ACCESS TOKEN ==

user -> fe : Gọi API có yêu cầu xác thực\n(e.g. GET /api/bookings/me)
fe -> ctrl : GET /api/bookings/me\nAuthorization: Bearer {accessToken}
ctrl -> jwt : Validate token
activate jwt

alt Token hết hạn
    jwt --> ctrl : TokenExpiredException
    ctrl --> fe : 401 Unauthorized
    fe -> user : "Phiên đăng nhập hết hạn.\nVui lòng đăng nhập lại"
else Token hợp lệ
    jwt -> jwt : Extract user info\n(userId, roles)
    jwt --> ctrl : Authentication principal
    deactivate jwt

    ctrl -> svc : Lấy thông tin user\n+ kiểm tra quyền
    activate svc
    svc --> ctrl : Dữ liệu booking của user
    deactivate svc

    ctrl --> fe : 200 OK + data
    fe -> user : Hiển thị danh sách booking
```

## 6. Các ràng buộc toàn vẹn riêng của phân hệ User

| STT | Ràng buộc | Mô tả |
|-----|-----------|--------|
| 1 | `UNIQUE(email)` | Mỗi email chỉ đăng ký một tài khoản duy nhất |
| 2 | `UNIQUE(phone_number)` | Số điện thoại không trùng lặp giữa các tài khoản |
| 3 | `(user_id, role_id)` trong user_roles | Không gán trùng vai trò cho cùng một user |
| 4 | `(user_id, target_type, target_id)` trong reviews | Mỗi user chỉ đánh giá một lần cho một đối tượng |
| 5 | `booking_id` trong reviews phải là COMPLETED | Chỉ cho phép đánh giá khi booking đã hoàn tất |
| 6 | `token_expires_at > NOW()` | Token xác thực/reset chỉ hợp lệ trong thời hạn |
| 7 | `failed_login_attempts < 5` khi đăng nhập | Khóa tài khoản khi sai quá 5 lần |
| 8 | `is_verified = TRUE` khi đặt tour | Tài khoản phải xác thực email trước khi đặt dịch vụ |
| 9 | `report.reporter_id ≠ review.user_id` | Người viết review không thể tự báo cáo review của mình |
| 10 | `ON DELETE CASCADE` cho user_roles | Xóa user sẽ xóa hết vai trò của user đó |
